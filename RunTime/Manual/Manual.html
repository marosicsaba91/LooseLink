LooseLink Service Locator

Functionality

What can LooseLink Services do and why to use it?

  1. LooseLink Services as a service locator can provide a (high level) object through static call regardless of the fact that the requested object is instantiated already or not.

    If it doesn't exist yet, it will be instantiated and cached automatically. Prefabs can find or create high level objects. Any object can get Information/Object it needs, even when it is not existing yet.

  1. Search is always based the objects type.

    The searches and its results can be interfaces or abstract types.

    Unwanted request types (like non abstract types) can even be blocked.

    Not necessary to search by name (which is highly fragile), but it is possible to specify the search through other properties than the type only. (Tags)

  1. It's setup, monitoring & debug is managed in the editor on a clear UI.

    Requires (almost) no coding.

    Every service can be found easily.

  1. Every requested object is initialized before it's provided by the service locator.

    It doesn't depend on the execution order of Unity event functions.

    Initialization happens only once.

    Cyclic reference is allowed.

    It's non-assync. Requested objects can be used instantly.

  1. Work in editor and play time too.

    Editor scripts can use it too.

  1. Keeps the project maintainable.

    It makes possible to easily change any module without breaking others.

    We want our code to be coupled by abstraction (interfaces).

    No need to reference concrete high level objects through [SerializeField].

    You can build a project where any prefabs can be instantiated in any scene and just work.

  1. Makes testing and other highly specific situations more easy to implement.

    The result of a request can change through situation. (For example testing)

    Possible to make mock versions of every module and use them without changing any of the (not test specific) software code.

  1. Request are efficient.

    No Reflection, No Memory Allocation in Playtime, Highly cached.

    Scales linearly with the possible services provided. Scaling doesn't depend on the complexity of the project any other way.

    Although it's as fast is I could make it, it's still recommended to use it mostly in Start & Awake.

Terminology

How we will refer something in the context of this tool?

Service Locator (SL): System that can provide

Service: Egy olyan magas szintű objektum, vagy annak egy intrface, amiből egy konkrét példány létezik egyszerre, vagy legalább egy paraméterezéssel csak egy példány létezik egyszerre.

Service Source: Egy objektum, ami képes valamiféle szabály szerint visszaadni (akár példányosítani is) egy vagy több Servicet-t.

Service Installer: Olyan objektum, amiken statikus lekérdezésen keresztül elérhetővé tudunk tenni ServicesSourcokat. Service Installer lehet MonoBehaviour komponens vagy ScriptableObject File.

Tag: Bármilyen C# objektum, amivel meg tudunk jelölni egy Service-t, hogy specifikusan le lehessen kérdezni, akkor is ha az adott típusú Service-ből több is létezik.

Manual

How to use LooseServices?

Minden szükséges kódot a LooseLink névtér tartalmaz.

A Loose Services Window a Tools / Loose Services menün kerszetül érhető el.

  • Service-ek definiálása

    Ahhoz, hogy egy objektumot le tudjunk kérni a SL-tól, az osztálynak vagy interface-nek meg kell valósítania az IService interface-t.

    Az IService interface-szel egy függvényt kell megvalósítani: void Initialize() { }

    Az Initialize függvény azelőtt meghívódik, hogy a SL átadná az objektumpéldányt a lekérdezőnek.

    Hasonló az Awake-hez, de nem aszinkron.

    Az Initialize függvény egyszer hívódik meg, (nem minden, csak az első lekérdezésnél).

    Initialize függvényben érdemes tartani minden SL lekérdezést, de más inicilaizációt is tartalmazhat.

    Különböző osztályok Initialize függvényei tartalmazhatnak körkörös egymásra hivatkozást is.

  • Service Source Installálása

    Service Source-ok lehetnek:

    • In Scene GameObject,
    • Prefab file, aminek gyökér GameObject-jén van olyan komponens, ami megvalósítja az IService Interface-t. (2 Beállítás: File és Prototype:)
      • Original Prefab File: A SL egy az eredeti prefabon lévő servicet adja vissza.
      • Prefab Prototype: A SL példányosítja a prefabot az aktív scenebe és a scene objekten lévő Service-t adja vissza. (Csak egy példányosítás)
    • ScriptableObject file, aminek osztálya megvalósítja az IService Interface-t. 2 Beállítás: File és Prototype.
      • Original File: A SL egy az eredeti ScriptableObject file-t adja vissza.
      • Prefab Prototype: A SL egy másolatot ad vissza
    • Nem abstract MonoBehaviour skript, ami az IService Interface-t. (Nem kell installer)
    • Nem abstract ScriptableObject skript, ami az IService Interface-t. (Nem kell installer)

    A service lekérdezője szempontjából láthatatlan, hogy az adott service milyen forrásból érkezik.

    Installerek: Az installerek olyan objektumok, amik Service source-okat tartalmaznak sorba rendezve.

    Global Installer-ek: Olyan Scriptable Object file-ok, amiken beállított Service Source-ok egész projektre érvényesek.

    Scene Installer-ek: Olyan Mono Behaviour Installerek, amiken beállított Service Source-ok akkor érhetők el a Sevice Locatoron keresztül, ha az Installer be van töltve és aktív a sceneben.

    Ha létezik egy nem-abstract MonoBehaviour vagy ScriptableObject IService osztály, akkor az példányosítható előzetes beállítás nélkül is. (Installer nélküli Service Source-ok)

    (Installer nélküli Service Source-hoz nem rendelhető semmi specifikus beállítás. Leinkább fallback opciókén javasolt használni.)

    Egy telepített Service Source lekérdezés mindig a legelső találatot adja vissza akkor is, ha több Source is megfelel a lekérdezés feltételének.

    Installerek és Service Source-ok sorrendje: (Legmagasabb prioritás felül)

    1. Később létrehozott Scene Installler Service Source-ai.
    1. Előbb létrehozott Scene Installler Service Source-ai.
    1. Global Installerek (Köztük a sorrend nem állítható.)
    1. Installer nélküli Service Source-ok

    (Installeren belül a Service source-ok sorrendje tetszőlegesen átrendezhető.)

    (Installer nélküli Service Source-hoz sorrendje nem átrendezhető)

    A telepített Sevice-ek sorrendje a Loose Services ablakban tekinthető meg.

    Egy Service source-ot hozzá lehet adni bármely installerhez annak inspector ablakán.

    Service Source Set: Service Source-ok kategorizálására szolgáló file.

  • Service Lekérdezése

    A service lekérdezése a következő statikus függvényekkel lehetséges:

    Services.Get<TService>();
Services.Get(Type iServiceType);
Services.Get<TService>(params object[] tags);
Services.Get(Type iServiceType, params object[] tags);

    A lekérdezés az első olyan Service-t adja vissza, ami megvalósítja a megfelelő típust és rendelkezik az össze átadott tag-gel.

    Egy lekérdezett típus lehet interface vagy absztrakt osztály is, de mindenképp le kell származnia az IService interface-ből.

    (Interface is tud örökölni Interface-től.)

    Ha le akarjuk tiltani egy IService típus lekérdezést, az adott típust a [IgnoreService] attribútummal kell megjelölni.

  • Servicek Taggelése

    A tagek arra szolgálnak...

    Ahhoz, hogy egy Service Source-ot tagekkel lássunk el, a nem-abszrakt típusának meg kell valósítani a az ITagged Interfacet, és vele a GetTags(); függvényt.

    Tag bármilyen típusú object lehet.

    something Completely Different

    (Ha a tag leszármazik az ITag interface-ből, akkor definiálni lehet a UI-on megjelenített színt és feltüntetettnevet.)

[SerializeField] TagFile[] tags;
public IEnumerable<object> GetTags()
{
    yield return 1; // integer
    yield return name; // string
    foreach (TagFile tagFile in tags) // any number of
        yield return tagFile; // Scriptable Object file
	  yield return (obejct) somethingCompletelyDifferent; // any type of object
}

Limits & deficiencies:

  1. Minden Service csakis MonoBehaviour vagy ScriptableObject lehet.

    Oka, hogy így minden service megtekinthető és szerkeszthető az Inspector ablakban.

  1. Mivel minden service-nek le kell származnia az IService interface-ből, nem lehet elérni olyan "külső" objektumokat, aminek a kódjához nincs hozzáférésünk. Pl.: EventManager Ilyen esetban érdemes az adott objektumot becsomagolni egy olyan saját szervice-be, amit elérünk. Pl.: class EvenetanagerProvider : IService
  1. Nem lehet generikus típust lekérdezni. Pl.: Sevvices.Get<BulletProvider<BasicBullet>>

    Sokkal lassab és komplikáltabb lenne a használata, ha ez a funkció támogatott lenne.

    Ebben az esetben érdemes egy üres specifikus típust definiálni, ami leszármozik a generikus osztályból és aztán azt lekérdezni.

    class BulletProvider : BulletProvider<BasicBullet> { }
